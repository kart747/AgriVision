Crops include tomato, apple and grape. We are going to stick to these crops in the next 5 years. Increasing scale has made crop disease detection a major operational challenge — impacting yield, cost, and long-term sustainability.

Challenge
Problem Statement

Mission Critical

Design and build an AI-powered crop disease intelligence system that analyzes leaf photographs using computer vision, drone/mobile imagery, and large language models, deployable as a local web application for real-time use by field agronomists.

Architecture
Expected End-to-End Pipeline
Stage 01
Drone Capture
Input layer — Field or leaf images from drone/mobile with GPS metadata attached.
Stage 02
Preprocessing
Transform — Resize, normalise, augment. Validate quality with blur detection checks.
Stage 03
CNN Model
Inference — EfficientNet / ResNet-based classification across disease classes.
Stage 04
Prediction
Output — Disease class, confidence score, and severity level for the detected condition.
Stage 05
LLM Module
Context layer — Recovery steps, treatment options, and geo-aware preventive guidance.
Stage 06
Web App UI
Presentation — Local web interface for image upload, results, and recommendation display.

All six stages must be functional and demonstrated live during judging. Partial pipelines will not be scored.

Technical Specification
System Requirements
Module 01

Image Processing Core

  • Input: Drone-captured field or leaf images :
  • Since Drone is out of the Hackathon Scope , you can assume we are getting pictures from Drone with Metadata
  • Output: Disease class with confidence score
  • Crops: Tomato, Apple, Grape
  • Dataset: Multi-class, includes healthy leaves and class imbalance (real-world simulation)
Module 02

AI Vision Model

  • Architecture: CNN
  • Transfer Learning: EfficientNet, ResNet or equivalent strongly recommended
  • Primary Metric: F1 Score
  • Also Required: Precision, Recall, Confusion Matrix (Good to have)
Module 03

Context-Aware LLM Layer

  • Inputs to LLM: Crop type, Disease detected
  • Outputs: Recovery steps, Treatment suggestions (organic & chemical), Time-to-recovery, Preventive measures
  • Approaches: Prompt Engineering - Use Free LLM * See the Note section below
Module 04 - Nice to have

Input Validation & Guardrails - Use your creativity and Innovation

  • Blur Detection: Flag low-quality images before inference
  • Confidence Threshold: Request re-capture if below 60%
  • GPS Validation: Warn if image location doesn't match farm region
Module 05 — Mandatory

Local Web Application (Minimum Viable Demo)

  • Frontend: Image upload, crop selection, prediction display, LLM recommendation panel
  • Backend: Model inference API running locally — Flask, FastAPI, Express, or Streamlit
  • Allowed LLM options: API-based rule-based prompt simulation * See note
  • No pure notebooks: Jupyter demos without a web UI will be disqualified

Quality Assurance
Input Validation Pipeline

A production-ready system rejects bad input before inference. Implement at least Layers 1–3:

Layer 1 · Critical
Image Quality Check — Detect blur using Laplacian variance. Reject images below threshold with "Image is unclear — please capture a closer leaf photo."
Layer 2 · High
Confidence Gate — If disease model confidence < 60%, flag and request re-upload rather than displaying an uncertain prediction as fact.

Rules
Disqualification Conditions

Any of the following will result in immediate disqualification

  • No working web application running on localhost during demo
  • Model not integrated into the UI — standalone Jupyter notebooks are not accepted
  • Hardcoded or mocked prediction outputs that bypass real model inference
  • LLM recommendations that are pre-written strings not generated from disease input
  • External paid APIs used without prior written justification submitted to organisers
  • Plagiarised code or models presented as original work
  • Code sourced or commissioned from external individuals, freelancers, or online services and presented as the team's own work
